У нас была проблема. Во время current topic диалогов, ретривер не понимал ответы Да, Давай и т.д.

Мы добавили LLM ноду перед ретриверов с таким промптом:

Ты переформулируешь запрос пользователя для поиска в базе знаний стоматологической клиники.

Правила:

- Если сообщение пользователя короткое или является согласием

("Да", "Ок", "Расскажи", "Давай") — посмотри на предыдущий

ответ ассистента и верни тему которую он предлагал продолжить.

- Если новый вопрос — верни его суть одной фразой.

- Возвращай ТОЛЬКО поисковый запрос. Без пояснений,

без кавычек, без точки в конце.

Этап 1. Ввести недостающие state-поля
Задача

Добавить в Flow State поля, которые будут явно хранить логику диалога.

Нужно добавить поля
1. last_offered_topic

Что бот последним предложил продолжить.

Примеры:

этапы имплантации

как проходит заживление

когда обратиться к врачу

2. last_offer_type

Тип последнего предложения бота.

Допустимые значения:

topic_followup

booking

general

3. covered_topics

Список тем, которые уже были раскрыты в рамках текущей темы.

Пример:

["почему не больно", "этапы имплантации"]
4. has_retrieved_content

Флаг: нашёл ли retriever вообще релевантный контент.

Допустимые значения:

true

false

5. has_current_topics

Флаг: есть ли у найденного контента подтемы для продолжения.

Допустимые значения:

true

false

6. query_mode

Режим, в котором была обработана текущая пользовательская реплика.

Допустимые значения:

new_question

topic_followup

booking_confirmation

neutral_short_reply

7. normalized_query

Нормализованный запрос, который реально уходит в retriever.

Что проверить после добавления

все поля создаются без ошибок;

state сохраняется между шагами диалога;

поля не затираются случайно на каждом run;

есть значения по умолчанию.

Этап 2. Переделать LLM-ноду перед Retriever
Цель

Эта нода должна не просто “переформулировать запрос”, а аккуратно определить тип реплики и вернуть структурированный результат.

Что поменять в логике ноды
Сейчас

Нода получает:

question

И возвращает:

просто строку для retriever

Нужно сделать

Нода получает:

question

last_offered_topic

last_offer_type

И возвращает JSON:

{
  "normalized_query": "string",
  "query_mode": "new_question | topic_followup | booking_confirmation | neutral_short_reply"
}
Что должна уметь эта нода
Сценарий 1. Новый вопрос

Если пользователь задал новый смысловой вопрос, нода:

кратко нормализует его;

ставит query_mode = new_question.

Пример:

вход: Сколько заживает десна после импланта?

выход:

{
  "normalized_query": "сроки заживления десны после имплантации",
  "query_mode": "new_question"
}
Сценарий 2. Короткое согласие на подтему

Если пользователь написал:

да

давай

расскажи

подробнее

и last_offer_type = topic_followup, тогда:

normalized_query = last_offered_topic

query_mode = topic_followup

Сценарий 3. Короткое согласие на запись

Если пользователь написал короткое согласие и last_offer_type = booking, тогда:

retriever не нужен;

normalized_query = ""

query_mode = booking_confirmation

Сценарий 4. Нейтральный короткий ответ

Если пользователь написал:

ок

понятно

ясно

угу

и это не новый вопрос, не follow-up и не подтверждение записи, тогда:

normalized_query = ""

query_mode = neutral_short_reply

Что важно отдельно прописать в промпте

не выдумывать тему, если она неочевидна;

не считать любое “да” записью;

не возвращать пояснения;

возвращать строго JSON;

не брать тему “из головы”, если нет опоры на state.

Что проверить

нода стабильно возвращает JSON;

short replies не превращаются в мусорные запросы;

Да после предложения подтемы идёт в тему;

Да после предложения записи не идёт в retriever.

Этап 3. Сделать лёгкую маршрутизацию по query_mode
Цель

Не все пользовательские реплики должны одинаково идти в retriever.

Нужная логика
Если query_mode = new_question

отправить normalized_query в retriever

Если query_mode = topic_followup

отправить normalized_query в retriever

Если query_mode = booking_confirmation

Если query_mode = neutral_short_reply

Варианты:

либо не запускать retriever вообще;

либо отвечать мягкой короткой репликой;

либо задавать один безопасный уточняющий вопрос.

Этап 4. Усилить Custom Function для current_topics
Цель

Сделать current_topics не просто строкой “как получилось”, а более управляемым слоем.

Что должна делать Custom Function
1. Проверять, был ли найден контент

Если retriever вернул чанки:

has_retrieved_content = true

Если нет:

has_retrieved_content = false

2. Извлекать подтемы

Из retrieved content:

искать заголовки ###

исключать ### Коротко

собирать список подтем

3. Ставить has_current_topics

Если после извлечения список непустой:

has_current_topics = true

Если подтем нет:

has_current_topics = false

4. Нормализовать формат current_topics

Лучше хранить не одной строкой через запятую, а списком.

Было:

Почему не больно, Как проходит операция, Что чувствуется после

Лучше:

["почему не больно", "как проходит операция", "что чувствуется после"]
5. Исключать уже покрытые темы

Если есть covered_topics, то доступные темы считать так:

available_topics = current_topics - covered_topics
Что проверить

current_topics собирается стабильно;

covered_topics реально вычитается;

повторно уже раскрытая тема не предлагается;

пустой current_topics не считается автоматически “информации нет”.

Этап 5. Развести 2 ситуации: “контента нет” и “подтем нет”
Цель

Не путать:

retriever ничего не нашёл;

retriever нашёл полезный контент, но без вложенных подтем.

Нужная логика
Сценарий A. has_retrieved_content = false

Бот должен:

честно сказать, что точной информации нет;

не выдумывать ответ;

не предлагать искусственные подтемы.

Сценарий B. has_retrieved_content = true и has_current_topics = false

Бот должен:

ответить по найденному контенту;

не говорить “информации нет”;

не строить пошаговый deep-dialog;

завершить ответ мягко и нейтрально.

Сценарий C. has_retrieved_content = true и has_current_topics = true

Бот должен:

ответить по сути;

предложить одну следующую подтему.

Что проверить

контакты, консультация, общие справочные ответы не ломаются;

reference-файлы без подтем не выглядят как “ничего не найдено”;

deep-файлы продолжают работать как пошаговая тема.

Этап 6. Сохранять last_offered_topic и last_offer_type после ответа бота
Цель

Чтобы следующий короткий ответ пользователя можно было правильно интерпретировать.

Что нужно сделать

После финального ответа бота система должна понимать:

Если бот предложил подтему

Пример:

Тогда сохранить:

{
  "last_offered_topic": "этапы имплантации",
  "last_offer_type": "topic_followup"
}
Если бот предложил запись

Тогда сохранить:

Если бот просто завершил ответ без конкретного оффера

Пример:

Тогда сохранить:

{
  "last_offered_topic": "",
  "last_offer_type": "general"
}
Важно

Не сохранять туда всю фразу бота.
Сохранять только смысловой ярлык предложения.

Этап 7. Начать вести covered_topics
Цель

Бот не должен повторно предлагать уже раскрытую тему.

Что нужно реализовать
После ответа бота

Если бот реально раскрыл одну тему из списка:

добавить её в covered_topics

Пример:

["почему не больно"]

После следующего раскрытия:

["почему не больно", "этапы имплантации"]
Что важно

Не отмечать тему как covered, если бот:

только мельком упомянул её;

перечислил её без раскрытия;

просто назвал как вариант продолжения.

Что проверить

бот не предлагает снова ту же подтему;

available_topics уменьшается по мере диалога;

когда темы закончились, бот не делает вид, что есть ещё 5 скрытых веток.
